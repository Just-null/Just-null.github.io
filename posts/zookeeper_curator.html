curator介绍

curator简介

curator是Netflix公司开源的一个zookeeper客户端，后捐献给apache， curator框架在zookeeper原生API接口上进行了包装，解决了很多zooKeeper客户端非常 底层的细节开发。提供zooKeeper各种应用场景(比如：分布式锁服务、集群领导选举、 共享计数器、缓存机制、分布式队列等)的抽象封装，实现了Fluent风格的API接口，是最 好用，最流行的zookeeper的客户端。

原生zookeeperAPI的不足:

  • 连接对象异步创建，需要开发人员自行编码等待
  • 连接没有自动重连超时机制
  • watcher一次注册生效一次
  • 不支持递归创建树形节点

curator特点：

  • 解决session会话超时重连
  • watcher反复注册
  • 简化开发api
  • 遵循Fluent风格的API
  • 提供了分布式锁服务、共享计数器、缓存机制等机制

使用curator连接zookeeper

curator的重连策略

  1. RetryUntilElapsed(int maxElapsedTimeMs, int sleepMsBetweenRetries)
  • 以sleepMsBetweenRetries的间隔重连,直到超过maxElapsedTimeMs的时间设置
  • 例如：RetryPolicy retryPolicy=new RetryUntilElapsed(10000,3000)，代表每3秒重连一次，总等待时间超过10秒后停止重连
  1. RetryNTimes(int n, int sleepMsBetweenRetries)
  • 以sleepMsBetweenRetries的间隔重连，重连n次
  • 例如：RetryPolicy retryPolicy = new RetryNTimes(3,3000)，每3秒重连一次，重连3次
  1. RetryOneTime(int sleepMsBetweenRetry)
  • 以sleepMsBetweenRetries的间隔重连，重连1次
  • 例如：RetryPolicy retryPolicy = new RetryOneTime(3000)，每3秒重连一次，重连1次
  1. ExponentialBackoffRetry
  • ExponentialBackoffRetry(int baseSleepTimeMs, int maxRetries)
  • ExponentialBackoffRetry(int baseSleepTimeMs, int maxRetries, int maxSleepMs)
    • 时间间隔 = baseSleepTimeMs * Math.max(1, random.nextInt(1 << (retryCount + 1)))

新增节点

create1()

  • 创建路径为/create/node1的节点
  • 节点权限列表为：world:anyone:cdrwa
  • 节点类型为：持久化节点
  • 创建方式：同步

以设置没命名空间（create）为根节点,如果不设置命名空间，则以/为根节点

create2()

  • 创建路径为/create/node2的节点
  • 节点权限列表为：ip授权，授权给10.211.55.5
  • 节点类型为：持久化节点
  • 创建方式：同步

create3()

  • 创建路径为/create/node3/node31的节点
  • 节点权限列表为：world:anyone:cdrwa
  • 节点类型为：持久化节点
  • 创建方式：同步

create4()

  • 创建路径为/create/node4的节点
  • 节点权限列表为：world:anyone:cdrwa
  • 节点类型为：持久化节点
  • 创建方式：异步

watcherAPI

curator提供了两种Watcher(Cache)来监听结点的变化

  • NodeCache : 只是监听某一个特定的节点，监听节点的新增和修改
  • PathChildrenCache : 监控一个ZNode的子节点. 当一个子节点增加， 更新，删除 时， Path Cache会改变它的状态， 会包含最新的子节点， 子节点的数据和状态

watcher1()

  • 监听/watcher1节点的变化

先启动watcher1，之后去创建节点，修改节点

  • 通过创建节点、修改节点，可以发现，curator提供的watcher一次注册生效多次

watcher2()

  • 监听以/watcher1为根路径的子节点的变化

tra1()

  • 没有使用事物

tra2()

  • 创建/node1节点
  • 更新/node2节点（node2节点并不存在）

代码执行完毕

分布式锁

  • InterProcessMutex：分布式可重入排它锁
  • InterProcessReadWriteLock：分布式读写锁

lock1()

  • InterProcessMutex：分布式可重入排它锁

启动两个进程，查看控制台输出

  • 通过观察可以发现，只有当进程1(先启动)释放了锁，进程2才能获取到对象

lock2()

  • InterProcessReadWriteLock：分布式读写锁
  • readLock()：获取读锁对象

启动两个进程，查看控制台输出

  • 通过观察可以发现，在进行写操作时，进程1和进程2可以并行执行

lock3()

  • InterProcessReadWriteLock：分布式读写锁
  • writeLock()：获取读锁对象

启动两个进程，查看控制台输出

  • 通过观察可以发现，在进行读操作时，只有当进程1(先启动)释放了锁，进程2才能获取到对象

在读操作的同时进行写操作

先执行lock2()然后执行lock3()

  • 通过观察可以发现，在读操作的同时进行写操作时，只有当lock2()(先启动,读锁)释放了锁，lock3()(写锁)才能获取到对象

在写操作的同时进行读操作

先执行lock3()然后执行lock2()

  • 通过观察可以发现，在写操作的同时进行读操作，只有当lock3()(先启动,写锁)释放了锁，lock2()(读锁)才能获取到对象
